## 2. 产品概述

- **2.1 产品名称与定位**
  - **产品名称**: 井然 (JingRan)
  - **产品定位**: 一款专为大学生和忙碌专业人士设计的，以智能任务规划为核心的轻量化、跨平台任务管理助手。
- **2.2 产品愿景与使命**
  - **愿景**: 成为用户最信赖的个人效率中枢，让每位用户都能毫不费力地掌控自己的时间和目标。
  - **使命**: 通过智能规划与无缝体验，将复杂的任务管理过程变得简单、有序且高效。
- **2.3 价值主张与独特卖点 (USP)**
  - **核心价值**: “井然”不仅是任务的记录工具，更是用户的“智能规划伙伴”。它能主动协助用户合理安排每日计划，轻松平衡长期目标与短期任务。
  - **独特卖点**:
    1.  **智能规划引擎**: 自动将任务插入日程，并能灵活应对突发事件，动态调整计划。
    2.  **完全本地化**: 所有数据本地存储和处理，确保隐私安全与离线可用性。
    3.  **无干扰体验**: 永久免费，无广告，专注于核心功能。
    4.  **跨平台无缝同步**: 支持手机、手表、电脑多端协同，数据通过本地网络（如蓝牙、Wi-Fi）同步。
- **2.4 目标平台列表**
  - **第一阶段**: Android (手机)
  - **第二阶段**: Wear OS (智能手表)
  - **第三阶段**: Windows, macOS, Linux (Ubuntu) (桌面端)
- **2.5 产品核心假设**
  - 用户需要的不仅是任务列表，更是如何安排这些任务的“策略建议”。
  - 用户愿意为了更好的规划能力和数据隐私，接受一个功能相对聚焦的应用。
  - 本地化的多设备同步方案能满足用户在主要场景下的使用需求。
- **2.6 商业模式概述**
  - 当前阶段为完全免费，不考虑盈利。未来可能探索为高级 AI 规划功能提供增值服务的模式。

## 3. 用户研究

- **3.1 目标用户画像**
  - **画像一：大学生 (小张)**
    - **人口统计特征**: 20岁，在校大学生，主修计算机科学。
    - **行为习惯与偏好**: 熟悉各类软件和应用，追求高效率。每天需要应对课程、作业、社团活动和个人学习计划（如背单词、学编程）。
    - **核心需求与痛点**: 
      - 痛点：课程表、作业DDL、长期学习目标（考研、背单词）和临时活动（社团会议）交织在一起，难以有效规划时间，常常顾此失彼。
      - 需求：需要一个能整合所有任务和日程，并能智能推荐每日学习计划的工具。
    - **动机与目标**: 目标是高效完成学业任务，同时不耽误个人长期发展计划，实现“work-life balance”。
  - **画像二：职场新人 (丽莎)**
    - **人口统计特征**: 25岁，互联网公司产品经理。
    - **行为习惯与偏好**: 工作节奏快，会议多，任务切换频繁。依赖数字工具管理工作和生活。
    - **核心需求与痛点**: 
      - 痛点：工作任务（需求评审、数据分析）和个人事务（健身、阅读）互相冲突。经常被突发会议或紧急需求打乱计划。
      - 需求：一个能帮助她保护“专注时间”，并能在计划被打乱后快速重新规划的工具。
    - **动机与目标**: 提升工作效率，确保关键项目按时推进，同时保障个人成长和生活质量。
- **3.2 用户场景分析**
  - **核心使用场景**: 
    1.  **学期初规划**: 小张通过Excel或图片识别功能，一键导入本学期的课程表到“井然”中。
    2.  **每日计划**: 早上，丽莎打开“井然”，应用已根据她的长期目标（如“完成项目A”）和今日日程，自动生成了建议的任务时间表。
    3.  **长期目标跟进**: 小张设定了“一年背5000个单词”的长期目标，“井然”将其分解为“每天背15个单词”的短期任务，并自动安排进每日计划。每完成一次，长期目标进度条随之更新。
    4.  **应对突发任务**: 下午，丽莎突然接到通知，一小时后要参加紧急会议。“井然”能让她快速插入这个“紧急任务”，并自动将原计划中受影响的任务顺延或建议在会后完成。
  - **边缘使用场景**: 
    - 在地铁上，小张通过手表查看今天的任务列表。
    - 晚上，丽莎在电脑上复盘今天的工作，并将未完成的任务拖拽到明天。

## 4. 市场与竞品分析

- **4.1 市场规模与增长预测**
  - 任务管理和效率工具市场成熟且竞争激烈，但用户对创新和细分场景的需求持续存在。
- **4.2 行业趋势分析**
  - AI赋能、跨平台协作、注重隐私和用户体验是当前的主要趋势。
- **4.3 竞争格局分析**
  - **直接竞争对手**: Todoist, 滴答清单 (TickTick), Microsoft To Do, Apple Reminders。
    - 优势：功能全面，生态成熟，云同步稳定。
    - 劣势：功能日益臃肿，大多采用订阅制收费或有广告，其规划功能较弱，偏向清单记录。
  - **间接竞争对手**: Notion, Trello (项目管理工具), Google Calendar (日程管理工具)。
- **4.4 竞品功能对比矩阵**

| 功能 | 井然 (JingRan) | Todoist | 滴答清单 |
| :--- | :--- | :--- | :--- |
| **智能规划** | **核心优势** | 较弱 (智能识别日期) | 有 (部分日历视图) |
| **本地优先** | **是** | 否 (云同步) | 否 (云同步) |
| **跨平台同步** | 本地网络同步 | 云同步 | 云同步 |
| **收费模式** | **完全免费** | 订阅制 | 订阅制 |
| **课程表导入** | **计划支持** | 否 | 否 |

- **4.5 市场差异化策略**
  - **专注智能规划**: 不做大而全的功能清单，将所有资源投入到打磨“智能规划引擎”上，成为用户的智能助手。
  - **捍卫用户隐私**: 以“本地优先”作为核心原则，吸引对数据隐私敏感的用户群体。
  - **极致性价比**: 通过免费无广告策略，降低用户使用门槛。

## 5. 产品功能需求

- **5.1 功能架构与模块划分**
  ```mermaid
graph TD
    A[井然 App] --> B[任务管理模块]
    A --> C[智能规划模块]
    A --> D[数据同步模块]
    A --> E[视图与提醒模块]

    B --> B1[任务CRUD]
    B --> B2[长期/短期任务]
    B --> B3[任务分解]
    B --> B4[课程表导入]

    C --> C1[自动排程算法]
    C --> C2[动态调整逻辑]
    C --> C3[任务优先级评估]

    D --> D1[设备发现]
    D --> D2[数据传输(蓝牙/Wi-Fi)]
    D --> D3[数据冲突解决]

    E --> E1[日历视图]
    E --> E2[列表视图]
    E --> E3[系统通知API集成]
  ```
- **5.2 核心功能详述**
  - **5.2.1 智能规划模块**
    - **功能描述**: 作为用户，我希望能在我设定好所有任务（包括固定的课程/会议、长期目标、短期任务）后，应用能自动为我生成一份合理的每日时间规划表，以便我能清晰地知道每个时间段该做什么。
    - **用户价值**: 节省用户自己规划任务的脑力消耗和时间成本，降低任务启动的心理门槛。
    - **功能逻辑与规则**:
      1.  **输入**: 用户的固定日程（如课程表）、所有待办任务（及其预估时长、截止日期、优先级）、用户设定的工作/学习时段。
      2.  **规划算法**: 
          - 优先在时间线上放置固定日程和高优紧急任务。
          - 在剩余的“空闲块”中，根据任务的优先级、截止日期和关联性（如长期目标的子任务），填充其他任务。
          - 允许用户为长期目标设定执行频率（如“每周三次”），算法将据此安排。
      3.  **动态调整**: 
          - 当用户插入一个“突发紧急任务”时，系统自动寻找可牺牲的低优任务或可顺延的任务，并给出调整建议。
          - 用户可以接受建议，或手动调整。
      4.  **效率统计**:
          - 基于当日已安排的任务（包括自动排程与手动插入）计算完成率和执行效率。
          - 数据展示包括：已完成任务数、已安排任务总数、今日完成率%。
    - **验收标准**:
      - [P0] 应用能够根据用户输入的任务和日程，生成一份无时间冲突的日计划。
      - [P0] 用户可以插入紧急任务，应用能自动顺延其他任务。
      - [P0] 系统展示当日效率统计（完成任务数/总已安排任务数、完成率%）。
      - [P1] 长期目标的子任务能被自动、周期性地安排进日计划中。

  - **5.2.2 任务管理模块**
    - **功能描述**: 作为用户，我想要轻松地创建、查看、编辑和完成我的任务，并且能将一个大的长期目标分解成若干个可执行的小任务，以便我能有条不紊地推进我的目标。
    - **用户价值**: 提供结构化、清晰的任务管理方式，让用户对所有任务和目标进展一目了然。
    - **功能逻辑与规则**:
      - 支持创建“长期任务”和“短期任务”两种类型。
      - “长期任务”下可以创建多个“子任务”。当所有子任务完成时，长期任务可以被标记为完成。
      - 支持从Excel文件或图片（通过OCR）识别课程表信息，并批量创建为固定日程事件。
    - **验收标准**:
      - [P0] 用户可以创建、编辑、删除、完成任务。
      - [P0] 用户可以创建长期任务，并为其添加/完成子任务。
      - [P1] 长期任务有明确的进度条，根据子任务完成情况更新。
      - [P2] 支持从图片或Excel成功导入课程表。

  - **5.2.3 数据同步模块**
    - **功能描述**: 作为用户，我希望我在手机上做的任何修改（如完成一个任务），都能自动同步到我的手表和电脑上，以便我在任何设备上看到的都是最新状态。
    - **用户价值**: 提供无缝的多端体验，让用户可以随时随地、使用最便捷的设备管理任务。
    - **功能逻辑与规则**:
      - 使用蓝牙或局域网Wi-Fi进行设备间通信。
      - 以时间戳作为版本控制，处理数据冲突时，默认保留最新修改的版本。
      - 后台服务应能定期检测可同步设备，并在数据发生变更时主动发起同步。
    - **验收标准**:
      - [P1] 在同一局域网下，手机和手表/电脑能自动发现并连接。
      - [P1] 在手机端完成一个任务后，10秒内手表和电脑端该任务状态变为已完成。

- **5.3 次要功能描述**
  - **提醒功能**: 用户可以为任务设置提醒，应用将通过调用系统通知API在指定时间推送通知。
  - **主题定制**: 支持深色/浅色模式切换。

- **5.4 未来功能储备 (Backlog)**
  - 更复杂的AI规划建议（例如，根据用户习惯推荐任务时间）。
  - 数据可视化报告（例如，每周效率分析）。
  - 与第三方日历（Google Calendar, Outlook）集成。

## 6. 用户流程与交互设计指导

- **6.1 核心用户旅程地图**
  ```mermaid
graph TD
    subgraph MVP
    A(下载安装) --> B(首次启动，引导设置);
    B --> C(创建第一个长期目标);
    C --> D(分解为几个子任务);
    D --> E(查看自动生成的今日计划);
    E --> F(完成一个任务);
    F --> G(收到任务完成的积极反馈);
    G --> H(添加一个临时任务);
    H --> I(查看计划动态调整);
    I --> J(设置任务提醒);
    end
  ```
- **6.3 对设计师 (UI/UX Agent) 的界面原型参考说明和要求**
  - **设计风格**: 严格遵循 Apple 的 iOS Human Interface Guidelines，即使是在 Android 和桌面端，也要保持统一的视觉和交互美学。控件、字体、间距、动画效果都应参考iOS原生应用。
  - **色彩**: 以深色调（Dark Mode）为主色调，提供一个舒适、沉浸的视觉环境。强调色可选用明亮、饱和的颜色以突出重要操作和信息。
  - **信息架构**: 首页应为“今日计划”视图，以时间轴形式展示，让用户一目了然。任务列表、长期目标等作为次级入口。
  - **操作焦点**: “添加任务”和“完成任务”应是最高频操作，其交互必须极致简化。

## 7. 非功能需求

- **7.1 性能需求**
  - **响应时间**: 界面切换和操作响应时间应小于200ms。
  - **启动速度**: 应用冷启动时间应小于2秒。
  - **资源使用率**: 后台同步服务在空闲时CPU占用率应低于1%。
- **7.2 安全需求**
  - **数据存储**: 所有用户数据必须加密存储在本地设备上。
  - **数据传输**: 本地网络同步过程中的数据需要进行加密传输。
- **7.3 可用性与可访问性标准**
  - 遵循 Material Design 和 iOS HIG 的可用性指南。
  - 关键按钮和文本需保证足够的对比度。
- **7.5 数据统计与分析需求**
  - （由于是本地应用，此项为可选）可在本地匿名化地记录关键事件，用于用户自行回顾效率。
  - **关键事件**: 任务创建数、任务完成数、计划调整次数、长期目标进展。

## 8. 技术架构考量

- **8.1 技术栈建议**
  - 考虑到跨平台（Android, iOS, Wear OS, Desktop）的需求，强烈建议使用 **Flutter** 或 **React Native** 等跨平台框架进行开发，以保证UI一致性和开发效率。
- **8.2 系统集成需求**
  - 需要调用操作系统的 **通知服务API** 来实现任务提醒。
  - 需要调用操作系统的 **蓝牙和Wi-Fi API** 来实现设备发现和数据同步。
- **8.3 技术依赖与约束**
  - **OCR**: 需要集成第三方本地OCR库（如 Tesseract.js 的移动端版本）来实现图片导入课程表功能。

## 10. 产品成功指标

- **10.1 关键绩效指标 (KPIs)**
  - **日均任务完成率**: （每日完成任务数 / 每日计划任务数）* 100%
  - **周活跃用户 (WAU)**: 每周至少打开应用并进行一次操作的用户数。
  - **智能规划采纳率**: （接受系统推荐计划的天数 / 总使用天数）* 100%
- **10.2 北极星指标定义与选择依据**
  - **北极星指标**: **有效规划天数 (Effective Planning Days, EPD)**
  - **定义**: 用户在一天内，既通过“井然”完成了至少一个任务，又创建了新任务或调整了未来计划的天数。
  - **选择依据**: 此指标不仅衡量用户的“活跃”，更衡量用户是否深度认同并使用产品的“规划”核心价值。它结合了执行（完成任务）和规划（创建/调整计划），反映了产品的核心用户循环。
- **10.3 指标监测计划**
  - 所有数据在本地进行计算和存储，应用内可以提供一个“我的效率”页面，向用户展示他们自己的指标，以保护隐私。
